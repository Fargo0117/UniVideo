## 2. 可行性研究的前提

### 2.1 要求

* **功能要求：**
* **基础业务：** 用户注册登录、视频上传（支持本地存储）、流媒体播放（基础 HTML5 播放器）。
* **浏览检索：** 首页视频分类导航（Tab切换）、关键词模糊搜索（过滤已发布状态）。
* **互动管理：** 视频点赞、收藏（Favorites）、评论（支持二级盖楼回复）。
* **个人中心：** 用户资料修改（头像/昵称/密码）、查看“我的投稿”、查看“我的收藏”。
* **后台管理：** 视频审核（通过/驳回）、全站视频检索与删除、管理员上传免审通道。
* **特色功能：** 上传时支持通过前端 Canvas 截取当前帧作为视频封面。


* **性能要求：** 数据库需针对高频查询（如评论树、收藏关系）建立索引优化；前端页面加载响应迅速。
* **安全与保密要求：** 严格的权限控制（RBAC），普通用户无法操作管理员接口；用户密码加密存储；防止未审核内容泄露。
* **完成期限：** 2周（14天）。

### 2.2 目标

* **管理目标：** 实现校园视频资源的集中化、合规化管理，替代传统的群文件传输模式。
* **技术目标：** 掌握前后端分离架构（Python Flask + Vue 3），实践复杂业务逻辑（如多表关联、权限分级、前端图像处理）的开发。
* **体验目标：** 提供流畅的 Web 视频浏览体验，支持个性化收藏与精准搜索。

### 2.3 条件、假定和限制

* **运行环境：** 建议使用 Windows 或 Linux 操作系统，部署 Python 3.8+ 及 MySQL 8.0 环境。
* **经费限制：** 零成本开发，全部使用开源技术栈。
* **硬件限制：** 视频及图片资源存储于服务器本地文件系统（`/static` 目录），受限于硬盘物理空间，不接入云对象存储。

### 2.4 进行可行性研究的方法

采用调查研究法和原型分析法。通过对比现有的文件传输方式（QQ群、网盘）与 Web 流媒体技术的优劣，评估新系统的价值；通过构建快速原型验证关键技术（如 Canvas 截图）的可行性。

### 2.5 评价尺度

* **开发成本：** 低（利用现有设备与开源软件）。
* **功能完备性：** 是否覆盖了从投稿、审核到消费、互动的全生命周期。
* **易用性：** 界面是否符合主流视频网站的操作习惯。

## 3. 对现有系统的分析

### 3.1 处理流程和数据流程

* **现状：** 视频文件散落在各个班级群、社团群或临时网盘链接中。
* **流程：** 拍摄者 -> 上传网盘/群 -> 复制链接 -> 粘贴分享 -> 观看者下载/转存 -> 本地播放。

### 3.2 工作负荷

* 人工管理群文件效率低下，历史文件难以查找，链接容易失效。
* 缺乏审核机制，管理员难以对违规内容进行追溯和处理。

### 3.3 费用开支

* 目前主要依赖免费网盘或社交软件，无直接金钱开支，但存在巨大的时间成本和管理成本。

### 3.4 人员

* 需群管理员人工维护文件，无专门的技术维护人员。

### 3.5 设备

* 依赖各自的个人电脑或手机。

### 3.6 局限性

* **检索难：** 无法通过关键词搜索特定内容的视频。
* **无互动：** 缺乏点赞、评论、收藏等社区反馈机制。
* **无监管：** 内容传播不可控，存在安全隐患。

## 4. 所建议的系统

### 4.1 对所建议系统的说明

UniVideo 是一个基于 B/S 架构的 Web 应用。

* **前端：** Vue.js 3 构建响应式界面，集成 Canvas 截图功能。
* **后端：** Python Flask 处理业务逻辑，提供 RESTful API。
* **数据：** MySQL 存储结构化数据，本地文件系统存储媒体文件。

### 4.2 处理流程和数据流程

* **投稿流程：** 用户选择视频 -> (可选)播放预览并Canvas截图生成封面 -> 提交 -> 系统存入本地并记录数据库（状态：待审核）。
* **审核流程：** 管理员后台查看待审列表 -> 预览视频 -> 点击通过/驳回 -> 更新数据库状态 -> 前台可见（如通过）。
* **互动流程：** 用户浏览 -> 搜索/分类筛选 -> 播放 -> 点赞/收藏/评论 -> 数据实时回写数据库。

### 4.3 改进之处

* **实时性：** 支持在线流媒体播放，无需下载等待。
* **便捷性：** 新增前端截图功能，简化了封面制作流程。
* **管理性：** 引入“先审后发”与“管理员特权”机制，既保证了安全又提升了官方运营效率。
* **个性化：** 增加了收藏夹和个人中心，满足用户对资源的整理需求。

### 4.4 影响

* **对设备的影响：** 服务器端需要预留足够的磁盘空间存储视频文件。
* **对用户的影响：** 用户需注册账号使用，操作界面直观，无需复杂培训。

### 4.5 局限性

* 由于未实现复杂的流媒体分片技术（如 HLS），在大并发或网络较差环境下，大文件的加载速度可能受限。
* 暂不支持高级播放功能（如弹幕、画质切换），互动形式相对基础。

### 4.6 技术条件方面的可行性

* **技术栈匹配：** 开发者已修读 Python 程序设计和 Web 开发相关课程，熟悉 Flask 框架、SQLAlchemy ORM 及 Vue 组件化开发。
* **关键技术验证：** Canvas 截图和 RBAC 权限控制已在原型阶段验证通过。
* **结论：** 技术上完全可行。

### 7.2 使用方面的可行性

系统界面参考主流视频网站设计，符合当代大学生的互联网使用习惯，易用性高，无需专门培训即可上手。
