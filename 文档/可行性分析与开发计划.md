## 1. 引言

### 1.1 编写目的

本文档旨在对“UniVideo 校园视界”系统进行全面的可行性研究与项目开发规划。通过对技术、经济、社会等方面的分析，论证项目的必要性与可行性，并制定详细的开发进度计划，为项目开发人员（软件工程231班 张某）及指导教师提供项目实施的依据 1。

- **待开发系统名称：** UniVideo 校园视界（校园视频分享平台）
- **任务提出者：** 软件工程课程设计指导教师
- **开发者：** 软件工程231班 张某
- **用户：** 全校师生、系统管理员
- **背景说明：** 当前校园内缺乏一个集中的、学生自主管理的视频资源分享平台。社团活动、课程录像、校园新闻等视频资料主要依赖网盘或第三方社交软件流转，存在检索困难、互动性差、缺乏个性化管理（如收藏、个人作品集）等问题。本系统旨在利用 Python 和 Web 技术构建一个功能完备的校园流媒体社区。

### 1.3 定义

- **UniVideo：** 本项目的系统名称。
- **Pre-audit（先审后发）：** 普通用户上传视频后，需经管理员审核通过才能在前台展示的机制。
- **Admin Privilege（管理员特权）：** 指管理员账号具备上传视频免审核（直接发布）以及对全站视频进行搜索和删除的管理权限。
- **Collection（收藏）：** 用户将感兴趣的视频保存至个人中心的行为。

## 2. 可行性研究的前提

### 2.1 要求

- **功能要求：**
  - **基础业务：** 用户注册登录、视频上传（本地存储）、流媒体播放。
  - **浏览检索：** 首页视频分类导航（Tab切换）、关键词模糊搜索（过滤已发布状态）。
  - **社区互动：** 点赞、收藏（Favorites）、评论（支持二级盖楼回复）。
  - **个人中心：** 用户资料修改（头像/昵称/密码）、查看“我的投稿”、查看“我的收藏”。
  - **后台管理：** 视频审核（通过/驳回）、全站视频检索与删除、管理员上传免审通道。
- **性能要求：** 数据库需针对高频查询（如评论树、收藏关系、审核状态）建立索引优化；支持毫秒级分类切换响应。
- **安全与保密要求：** 严格的权限控制（RBAC），普通用户无法操作管理员接口；密码加密存储。
- **完成期限：** 2周（14天）。

### 2.2 目标

- **管理目标：** 实现校园视频资源的集中化、合规化管理。
- **体验目标：** 提供接近主流视频网站（如B站）的流畅体验，包括搜索、收藏和个性化主页。
- **技术目标：** 掌握前后端分离架构，实践复杂业务逻辑（如多表关联、权限分级）的开发。

### 2.3 条件、假定和限制

- **运行环境：** Windows/Linux，Python 3.8+, MySQL 8.0+。

- 

  **硬件限制：** 视频及图片资源存储于服务器本地文件系统 (`/static`），不依赖云存储 3。

  

  

### 2.4 进行可行性研究的方法

采用原型法与对比分析法。基于当前校园网盘分享的痛点，构建 Web 原型进行功能验证。

### 2.5 评价尺度

- **系统完整性：** 是否覆盖了从投稿、审核到消费、互动的全生命周期。
- **交互友好度：** 搜索、收藏、个人中心等功能是否符合用户直觉。

## 3. 对现有系统的分析

### 3.1 处理流程和数据流程

- **现状：** 视频文件散落在各个群文件或网盘链接中。
- **缺陷：** 无法搜索特定内容，无法收藏以便日后观看，无法建立个人作品集，管理员无法统一监管内容。

### 3.2 工作负荷

人工审核群文件效率低下，且无法追溯历史违规记录。

### 3.3 局限性

现有方式缺乏“社区”属性，仅仅是文件的搬运，缺乏点赞、评论、收藏等激励创作者的反馈机制。

## 4. 所建议的系统

### 4.1 对所建议系统的说明

UniVideo 采用 Python Flask + Vue 3 前后端分离架构。前端负责页面交互与路由管理，后端负责 RESTful API 接口与数据持久化。

### 4.2 处理流程和数据流程

- **智能上传流程：**
  - **普通用户：** 上传 -> 状态“待审核” -> 管理员后台通过 -> 发布。
  - **管理员：** 上传 -> 系统识别角色 -> 自动设置状态“已发布” -> 前台立即可见。
- **检索与互动流程：**
  - 用户输入关键词/点击分类 -> 后端过滤 `status=1` 的数据 -> 返回列表。
  - 用户点击收藏 -> 系统检查 `collections` 表 -> 若未收藏则写入，若已收藏则删除 -> 更新状态。
- **管理流程：**
  - 管理员进入后台 -> 搜索/筛选视频 -> 执行审核或删除操作 -> 数据库物理/逻辑删除。

### 4.3 改进之处

- **功能全面性：** 新增了**搜索**和**分类**，解决了“找视频难”的问题；新增了**个人中心**和**收藏**，解决了“视频留存难”的问题。
- **管理高效性：** 赋予管理员免审特权和全站搜索删除能力，极大提升了运维效率。

### 4.4 影响

- **对用户的影响：** 用户拥有了专属的个人空间，可以管理自己的创作和收藏。
- **对开发的影响：** 需设计更复杂的数据库模型（如 `User-Video` 的多对多收藏关系）。

### 4.5 技术条件方面的可行性

开发者已掌握 Flask 蓝图（Blueprint）、SQLAlchemy 关联查询及 Vue 组件化开发技术，完全能够实现上述高级功能 4。



## 5. 可选择的其他系统方案

### 5.1 方案一：购买商业视频点播系统

- **优点：** 功能强大，稳定性高。
- **缺点：** 费用昂贵，不适合课程设计，无法锻炼编码能力。

### 5.2 方案二：基于 Bilibili 等公网平台

- **优点：** 无需开发。
- **缺点：** 无法限制在校内访问，无法自定义审核标准，数据不掌握在自己手中。

### 10.5 关键问题

- **多表关联查询效率：** 在个人中心查询“我收藏的视频”时，需要 `users` -> `collections` -> `videos` 的多对多查询，需使用 SQLAlchemy 的 `secondary` 关联优化。
- **权限边界控制：** 必须确保普通用户无法调用 `/api/admin` 下的接口，需在后端通过装饰器严格校验 `current_user.role`。
