## 1. 引言

本文档旨在对“UniVideo 校园视界”系统进行全面的可行性研究与项目开发规划。通过对技术、经济、社会等方面的分析，论证项目的必要性与可行性，并制定详细的开发进度计划，为项目开发人员（软件工程231班 张某）及指导教师提供项目实施的依据 3。

## 2. 可行性研究的前提

### 2.1 要求

- 

  **功能要求：** 实现用户注册登录、视频上传（本地存储）、视频分类展示、视频播放、互动评论（支持二级回复）、点赞以及管理员后台审核视频等核心功能 5。

  

  

- **性能要求：** 系统需支持常规 MP4 格式视频的流畅播放，数据库需针对高频查询（如评论树、审核状态）建立索引优化。

- **安全与保密要求：** 必须建立严格的视频审核机制，防止不良信息传播；用户密码需加密存储。

- **完成期限：** 2周（14天）。

### 2.2 目标

- **管理目标：** 实现校园视频资源的集中化、数字化管理。
- **效率目标：** 相比传统网盘分享，视频检索和播放的效率显著提高。
- **技术目标：** 掌握前后端分离开发模式（Python Flask + Vue）及数据库优化设计。

### 2.3 条件、假定和限制

- **运行环境：** 建议使用 Windows 或 Linux 操作系统，部署 Python 3.8+ 及 MySQL 8.0 环境。

- **经费限制：** 零成本开发，使用开源技术栈。

- 

  **硬件限制：** 视频文件存储于服务器本地硬盘（/static 目录），受限于硬盘物理空间，不接入云对象存储 6。

  

  

### 2.4 进行可行性研究的方法

采用调查研究法和原型分析法。通过对比现有的文件传输方式（QQ群、网盘）与 Web 流媒体技术的优劣，评估新系统的价值。

### 2.5 评价尺度

- 开发成本低（开源免费）。
- 功能完备性（覆盖上传、审核、播放全流程）。
- 用户操作的简易程度。

## 3. 对现有系统的分析

### 3.1 处理流程和数据流程

目前校园视频分享主要依靠“人工传输”或“网盘链接”：

- **流程：** 拍摄者 -> 上传网盘 -> 生成链接 -> 复制到QQ/微信群 -> 观看者下载/转存 -> 观看。
- **局限：** 流程繁琐，数据分散，无法直接在线预览，且无法进行公开的点赞和评论互动。

### 3.2 工作负荷

随着校园活动增加，视频文件数量激增，人工管理群文件的负荷巨大，且历史文件难以查找。

### 3.3 局限性

现有方式缺乏审核机制，容易传播违规内容；缺乏分类索引，查找特定视频（如某门课的录像）如同大海捞针。

## 4. 所建议的系统

### 4.1 对所建议系统的说明

UniVideo 是一个基于 B/S 架构的 Web 应用。

- **前端：** 使用 Vue.js 构建响应式界面，模仿主流视频网站风格。
- **后端：** 使用 Python Flask 框架处理业务逻辑。
- **数据：** 使用 MySQL 存储元数据，本地文件系统存储实体文件。

### 4.2 处理流程和数据流程

- **投稿流程：** 用户填写信息并上传 -> 系统存入本地并记录数据库（状态：待审核） -> 管理员后台查看 -> 审核通过 -> 状态更为“已发布” -> 前台展示。
- **评论流程：** 用户发起评论 -> 系统判断是主评还是回复 -> 自动关联 Root ID 以优化查询 -> 存入数据库。

### 4.3 改进之处

- **实时性：** 支持在线流媒体播放，无需下载。
- **互动性：** 增加了点赞（含点赞时间记录）和多级评论功能，构建社区氛围。
- **安全性：** 引入“先审后发”机制，确保内容合规。

### 4.4 影响

- **对设备的影响：** 服务器端需要预留足够的磁盘空间存储视频文件。
- **对用户的影响：** 用户需注册账号使用，操作界面直观，无需复杂培训。

### 4.5 技术条件方面的可行性

- **开发技术：** 开发者已修读 Python 程序设计和 Web 开发相关课程，熟悉 Flask 框架和 SQL 语句，具备开发此系统的能力。

- **硬件支持：** 个人笔记本电脑（16G内存以上）足以支撑开发和本地测试环境。

- 

  **结论：** 技术上完全可行 7。
